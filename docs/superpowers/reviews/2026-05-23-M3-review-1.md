# M3 plan — review cycle 1

**Reviewer:** independent peer-reviewer subagent

## Summary

The plan is strong overall: the file map matches SHARED_CONTEXT.md §2 exactly, the seam types are reproduced verbatim, the only-`internal/iosbackend/`-imports-go-ios rule is honoured, TDD cadence (RED → verify-RED → GREEN → verify-GREEN → commit) is consistently applied, every commit step ends with the "Wait for explicit user approval" gate, `git add` always targets explicit paths, NZ English is used throughout, and the destructive surface is correctly deferred to M4. The four open questions are surfaced at the top rather than buried.

However, three substantive issues need attention before execution: (1) `runCrashLogsPull` calls `Client.Pull` *per entry* with a basename pattern, and the `crashlogs.FakeClient.Pull` ignores the result's `Bytes` accumulation in a way that diverges from the seam's "Pull is the bulk call" contract; (2) the per-entry `Pull(pattern=basename)` strategy effectively re-runs `List` + `WalkDir` on the device once per entry inside the iosbackend adapter, producing N+1 transport round-trips and reopening the AFC connection N times — the plan acknowledges the trade-off but understates the cost; (3) compile-time guard lines (`var _ = path.Clean`, `var _ = errors.New`, three more in Task 6) are deliberately written into committed code to mask unused imports — this fails `go vet`/`gofmt` cleanliness Task 8 Step 16 asserts and is a poor pattern to commit, even temporarily. There are also smaller findings around the duplicated `destPath`/`destPathHost` helper, the chained-fake type embedding pattern, and the goroutine leak on context cancellation in `stdinPrompter`.

## Findings

### Critical

None. M3 is read-only on the device, no go-ios imports leak outside `internal/iosbackend/`, and the safety + approval semantics are intact.

### High

1. **[High] Per-entry `Client.Pull(pattern=filepath.Base(e.Path))` re-runs the full crash-report walk + reopens AFC for every file.** (Task 7 Step 3, lines ~1664; Task 7 plan prose, lines ~1535–1540.) The iosbackend adapter's `Pull` (Task 5, lines ~1025–1061) itself calls `c.List(ctx, udid, pattern)` → `crashreport.ListReports(device, "*")` → walks the whole crash-report tree on the device, then opens a fresh AFC connection via `openCrashReportAfc(device)`. Doing this once per matched entry in `runCrashLogsPull` means 50 crashes → 50 full device walks + 50 AFC connections + 50 mover-flush round-trips (`ListReports` calls `moveReports` every time). On a phone with hundreds of `.ips` files this will be noticeably slow and possibly trigger the rate-limiting suspected in RESEARCH.md §7 question 2. Recommended fixes (pick one, no seam change needed):
   - Inside `runCrashLogsPull`, call `deps.Client.List` once, drive the loop over the entries, and call `deps.Client.Pull(ctx, udid, *patternFlag, *outFlag)` exactly once **after** the prompt phase — *but* this still needs to honour the user's per-file no answers. The simplest implementation: after gathering the user's decisions, fall back to filename-equality patterns only for the surviving subset and accept the inefficiency, OR
   - Add a per-entry `PullOne(ctx, udid, srcPath, dstPath)` to the seam. This *would* be a SHARED_CONTEXT.md §3 change, so surface it as an Open question and let the user decide.
   - At minimum: refactor `crashLogsClient.Pull` so it accepts an already-listed slice through an unexported fast path, and cache the AFC connection per call.

   This is a correctness-adjacent performance issue: on a real device it may not just be slow, it may fail. The plan's own RESEARCH.md §7 question 2 flags this exact risk. Recommend revising before execution.

2. **[High] `var _ = path.Clean`, `var _ = errors.New`, `var _ = os.MkdirAll`, `var _ = filepath.Dir`, `var _ = strings.TrimLeft` are committed to silence unused-import errors during scaffolding.** (Task 5 Step 3, lines ~1104–1108; Task 6 Step 3, lines ~1395–1397.) Task 8 Step 16 asserts `go vet ./... && gofmt -l .` are clean — but `goimports` would strip these imports entirely, and `go vet` does flag `var _ = foo` blocks as a code smell in some linters. More importantly, this is the "Steps that describe what to do without showing how" anti-pattern from SHARED_CONTEXT.md §7's forbidden-plan-patterns list — the plan tells the implementer to "delete unused imports if `goimports` requires it" rather than showing the correct minimal set. **Fix:** drop the placeholder imports entirely. In Task 5, omit `path` and `errors` from the import block (they're not used by the scaffold). In Task 6's stub `runCrashLogsPull`, just `fmt.Fprintln(stderr, "...")` and `return 2` — no need to reference `os.MkdirAll`/`filepath.Dir`/`strings.TrimLeft`. Then Task 7's diff adds the imports cleanly.

3. **[High] Pattern-pass-through to `crashreport.ListReports`.** (Task 5 Step 3, line ~992: `crashreport.ListReports(device, "*")`.) The plan correctly chooses to pass `"*"` and filter host-side via `MatchEntries`, which is defensible. But go-ios's `ListReports` *itself already filters by `filepath.Match(pattern, filepath.Base(path))`* during its `WalkDir` (verified at https://raw.githubusercontent.com/danielpaulus/go-ios/main/ios/crashreport/crashreport.go lines ~71–95). The plan's host-side filter is therefore correct but operationally redundant when the user supplies a non-`*` pattern. This is not a bug — it's defensive and Open question #3 documents the trade-off — but the plan's stated reason ("pattern semantics are predictable across go-ios versions") is overstated: the actual semantics ARE identical. Recommend either passing the user's pattern through and dropping the redundant host filter, or keeping the host filter and updating Open question #3 to say "we filter twice; the second filter is a no-op for the current go-ios version, defensive for future versions". As written the comment misleads a future reader.

### Medium

4. **[Medium] Goroutine leak on context cancellation in `stdinPrompter.Confirm`.** (Task 2 Step 3, lines ~322–350.) When `ctx.Done()` fires, `Confirm` returns immediately but the goroutine that called `br.ReadString('\n')` is still blocked on the underlying `io.Reader`. For `os.Stdin` (the real wiring), the goroutine survives until the process exits or the user hits Enter. For a long-lived process this would be a leak. For this short-lived CLI it's harmless. The plan does NOT explicitly acknowledge this, and the `TestStdinPrompter_Confirm_respectsContextCancellation` test uses `t.Cleanup(func() { close(br.done) })` which masks the leak in tests. **Recommend:** add a comment in the implementation noting "the read goroutine cannot be cancelled; it dies when the process exits or stdin closes; acceptable for a one-shot CLI". This is an integrity issue not a bug.

5. **[Medium] `destPath` and `destPathHost` are byte-identical and deliberately duplicated.** (Task 5 Step 3, lines ~947–951 vs Task 7 Step 3, lines ~1688–1691.) The justification ("avoid the cmd layer importing iosbackend which would drag go-ios into every unit test binary") is sound, but the duplication will rot. A cleaner option: put `destPath` in `internal/crashlogs/crashlogs.go` (it's a pure path helper with no go-ios dependency), then both the iosbackend adapter and the cmd layer import it from there. The crashlogs package already owns `MatchEntries` for similar reasons. **Recommend:** move the helper into `internal/crashlogs/`.

6. **[Medium] Skipped-overwrite counted as a failure may surprise users.** (Task 7 Step 3, lines ~1657–1660, and Open question #4.) The author calls this out as a deliberate choice but it's worth a UX note: the summary line `"pulled 1 of 2, ..., 1 failures"` and the non-zero exit code conflate "the device returned an error" with "the user declined". A user running `ios-tidy crashlogs pull` over a re-used out-dir would see scary failure language for what's actually working-as-intended skip behaviour. **Recommend:** split the summary into `"pulled X of Y, skipped Z (declined), failed W"` and consider mapping `--force`-able skips to a different exit code (e.g. 3) so scripts can distinguish.

7. **[Medium] `chainedFakeClient` embeds `*crashlogs.FakeClient` and overrides `Pull`, but recording goes to the embedded struct's `PullCalls` slice via `c.PullCalls = ...`.** (Task 7 Step 9, lines ~1793–1806.) This works in Go but is fragile: any test that introspects `c.FakeClient.PullCalls` vs `c.PullCalls` could get inconsistent results if `Pull` is also called on the embedded pointer. **Recommend:** rather than embedding, make `chainedFakeClient` its own type with its own recording slice, or extend `FakeClient` itself to support a `PullResults []PullResult` chain (which keeps the seam in one place).

### Low

8. **[Low] `ListDevices`/`DeviceList.DeviceList` access pattern.** (Task 5 Step 3, lines ~957–965.) The plan acknowledges the API may have changed at later pins; that's appropriate but the code as written assumes `ios.ListDevices()` returns a struct with a `DeviceList` field of `[]ios.DeviceEntry`, and each entry exposes `Properties.SerialNumber`. Worth a quick verification step against the pinned v1.0.213 source before Task 5 Step 5's `go build ./...`. If the field name differs, the implementer will see the build error — minor risk.

9. **[Low] Acceptance criterion text uses "mtime" but the plan renders `-`.** SHARED_CONTEXT.md §8 says "lists entries with path, size, mtime". The plan's compromise (zero-time → `-` rendering) is acceptable given the verified go-ios limitation, but SHARED_CONTEXT.md is currently unaware of the compromise. **Recommend:** the human user updates SHARED_CONTEXT.md §8 to read "path, size, mtime (when available)" so future M3 reviewers don't flag this. The plan's Open question #1 is the right level of escalation; this is just a docs nit.

10. **[Low] `failureOf` always sets both `Err` and `ErrMsg`.** (Task 5 Step 3, lines ~1096–1102.) Minor: when `err == nil` it returns a `Failure` with empty `ErrMsg` and `Err == nil`, which would never be added in practice. Harmless but the nil-check guard is dead code.

11. **[Low] `contains` helper in `prompt_test.go`.** (Task 1 Step 5, lines ~178–185.) Just use `strings.Contains` — already imported in Task 2's tests. Saves a hand-rolled substring search.

12. **[Low] `errors.Is(res.err, io.EOF)` check.** (Task 2 Step 3, line ~340.) `bufio.Reader.ReadString` returns plain `io.EOF`, not a wrapped error, so `errors.Is` works but `==` would be equivalent and clearer. Style only.

## Acceptance-criteria assessment

Walking each SHARED_CONTEXT.md §8 M3 bullet:

- **"`crashlogs list [--device UDID] [--pattern GLOB] [--json]` lists entries with path, size, mtime."** — **Partially met.** Path: yes. Size: yes (via per-path `Stat`). mtime: **not met as written** — the adapter sets `time.Time{}`. The plan's workaround (zero-time + `-` rendering, JSON `"0001-01-01T00:00:00Z"`) is acceptable because the go-ios v1.0.213 source genuinely drops `st_mtime` (verified at https://raw.githubusercontent.com/danielpaulus/go-ios/main/ios/afc/client.go lines 293–305 — only `st_ifmt`, `st_size`, `st_mode`, `st_linktarget` are parsed). My verdict: **accept the workaround**, recommend updating SHARED_CONTEXT.md §8 to "mtime (when available)" so the acceptance criteria reflect platform reality. The alternative — vendoring a fork of go-ios to add `st_mtime` parsing — is wildly out of scope for M3 and the cost/benefit is poor. The third alternative — using `crashreport.ListReports`' internal `WalkDir` to capture sizes (and any future mtime) in one pass instead of N+1 — is a worthwhile optimisation but unrelated to the mtime question.

- **"`crashlogs pull --out DIR [--device UDID] [--pattern GLOB]` copies matching entries to DIR preserving relative paths under DIR. Reports counts + bytes. Returns non-zero on any failure but proceeds through all entries."** — **Met** in shape; **at risk** on the "proceeds through all entries" semantics because the per-entry `Pull` strategy means a transport failure on the *list* phase aborts the whole pull rather than recording per-entry failures. The unit tests pin the happy path and the partial-failure path via the `chainedFakeClient`, but the real iosbackend `Pull` returns `(PullResult{}, err)` if `findDevice` or `openCrashReportAfc` fail mid-stream. See High finding #1.

- **"Default pattern: `*`. Pattern is `filepath.Match` semantics (single-segment)."** — **Met.** Both `runCrashLogsList` and `runCrashLogsPull` set `--pattern` default to `"*"`. `MatchEntries` applies `filepath.Match(pattern, filepath.Base(path))`, which is single-segment by construction. Multi-segment patterns like `subdir/*.ips` would never match because `filepath.Base` strips the directory — the plan does not explicitly test this but the behaviour is correct.

- **"Out dir created if missing; existing files overwritten with confirmation via `Prompter` unless `--force`."** — **Met.** `os.MkdirAll(*outFlag, 0o755)` and the overwrite-prompt-no test (`TestCrashLogsPull_overwritePromptNo_*`) plus the `--force` test (`TestCrashLogsPull_forceFlag_*`) cover this.

- **"Unit tests cover: pattern filtering, total bytes math, overwrite confirmation flow, partial-failure reporting."** — **Met.** Task 8 Step 4's acceptance grid maps each item to a named test. Total bytes math is exercised only inside the partial-failure test; a dedicated test would strengthen this but is not strictly required.

- **"`//go:build device` integration test pulls into a temp dir."** — **Met.** `TestDevice_CrashLogs_ListAndPullIntoTempDir` in `internal/iosbackend/crashlogs_device_test.go`, skips on missing `IOS_TIDY_TEST_UDID`, calls `Pull` into `t.TempDir()`, asserts no error. Per SHARED_CONTEXT.md §5 destructive guards: the test does NOT call `Remove`, which is correct for M3.

## Open questions raised by the plan author

The plan surfaces four open questions at the top. My verdict on each:

1. **mtime via `afc.Client.Stat` returns zero.** **Accept the workaround.** Verified the claim against go-ios `main` HEAD: `Stat` parses only `st_ifmt`, `st_size`, `st_mode`, `st_linktarget`. The `st_mtime` value IS sent by the device-side AFC daemon (per Apple's AFC protocol documentation, key names match) but the Go client drops it silently. The plan's chosen workaround (`time.Time{}` + `-` rendering + JSON zero-value) is the minimum-pain path. Alternatives considered: (a) vendoring a forked go-ios — out of scope, large maintenance cost for one field; (b) sending an upstream PR — recommended as a follow-up but not blocking M3; (c) updating SHARED_CONTEXT.md §8 to relax the criterion — recommended (see Low finding #9). **No revision needed for execution.**

2. **`crashreport.ListReports` returns paths only; opening our own AFC.** **Accept.** Verified: `ListReports` constructs `afcClient := afc.NewFromConn(deviceConn)` internally but does NOT return it, so the plan's `openCrashReportAfc` helper is genuinely necessary. The service name (`com.apple.crashreportcopymobile`) matches go-ios verbatim. One efficiency note: `ListReports` itself does a `WalkDir` over the entire tree, so doing a second `WalkDir` (via per-path `Stat` calls) is duplicate work. A more efficient design would be to inline a custom `WalkDir` that captures size during the walk. But that's an optimisation, not a correctness issue. **No revision needed for execution; consider as a follow-up.**

3. **Pattern semantics: pass `"*"` and filter host-side.** **Accept with caveat.** Verified that `crashreport.ListReports`' pattern argument is already `filepath.Match(pattern, filepath.Base(path))` (lines 84–86 of crashreport.go). So the plan's approach is correct but the stated justification ("predictable across go-ios versions") is overstated — the current behaviour IS identical. The host-side filter is defensive against future go-ios changes. See High finding #3.

4. **Skipped-overwrite counted as a failure → non-zero exit.** **Accept as defensible, raise UX concern.** The author's binding interpretation is defensible (any deviation from "all entries successfully pulled" is non-zero). I'd argue it should be split into a distinct "skipped" category for clarity. See Medium finding #6. **No revision required for execution if the human user agrees with the strict interpretation.**
