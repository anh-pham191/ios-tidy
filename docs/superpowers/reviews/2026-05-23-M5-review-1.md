# M5 plan — review cycle 1

**Reviewer:** independent peer-reviewer subagent

## Summary

A thorough, well-structured plan that lands the M5 evidence layer
(`apps list` + `apps probe`) honestly: probes are sequential, timeouts
classify as `ProbeError` (not `ProbeRefused`), persistence is atomic
via temp-file + rename, and `ProbeRefused` / `ProbeError` are exit-0
outcomes. The classification regex set covers the load-bearing cases
from RESEARCH.md §3 / go-ios #653 (`VendContainer.*failed`,
`connect afc service failed`) case-insensitively. The dual-pass
"compile then race-detect" structure for atomic writes is a nice
touch. Two High findings cluster around Task 6 (deliberately messy
intermediate code shown before the "use this final form" note —
risk that an executor commits the wrong block) and Task 12 (the
`store := c.deps.Store` "cleaner replacement" muddles which block
ships). One Medium concerns a transport error that happens to
contain the word "denied" being mis-classified as `ProbeRefused`.
Everything else is Low / nitpicks. Overall I'd approve with the
listed minor fixes; nothing is gating.

### High

1. **Task 6 ships two competing implementations of the timeout
   branch.** The Step-3 block introduces `timeoutBudget(ctx)`, then
   a paragraph below says "Use [the final form] — drop the unused
   helper." The plan format requires a single, copy-pastable
   implementation per step (§7 "Steps that describe what to do
   without showing how" is a forbidden pattern; the inverse —
   showing two and asking the executor to choose — is the same
   class of defect). An executor running on autopilot may commit
   the first block including the dead `timeoutBudget` helper and
   the unused `fmt` import. **Recommendation:** delete the first
   code block in Task 6 Step 3 and keep only the final `"timeout: "
   + err.Error()` form. (`plans/2026-05-23-M5-apps-probe.md:735-806`)
   See `code-review.md` §"Constructive Feedback" (offer one
   solution) and §7-forbidden-patterns.

2. **Task 12 has the same shape problem in the store-resolution
   block.** Step 3 shows an `if store == nil { ... } else if
   *storeDir != "" { _ = dir }` block whose `else-if` is admittedly
   a no-op, then a paragraph below says "Use [the cleaner block]
   and discard the inline else-if." Same risk as Task 6. Compounding
   it, the `_ = dir` line references a variable not in scope at
   that point — the code as written would not compile. **Recommendation:**
   show only the cleaner block. (`plans/2026-05-23-M5-apps-probe.md:1978-2096`)

### Medium

1. **A transport error containing "denied" by coincidence is
   mis-classified as `ProbeRefused`.** The regex
   `(?i)denied|refused|vendcontainer.*failed|connect afc service failed`
   matches any error whose message contains "denied" in any
   context — e.g. a hypothetical lockdown error
   `"pair-record path denied by TCC"` (RESEARCH.md §6, issue #710)
   would be classified `ProbeRefused` rather than `ProbeError`.
   This matters because M6 will refuse to clean `ProbeRefused`
   apps but offers retry hints for `ProbeError` — a TCC-blocked
   user would be told "policy refusal, use Settings" when the real
   answer is "fix your TCC pair-record". **Recommendation:** either
   anchor the daemon-refusal pattern to the `VendContainer` /
   `house_arrest` context (e.g.
   `(?i)(vendcontainer|house_arrest|afc service).*(denied|refused|failed)|connect afc service failed`),
   or add a `reTransport` pattern that wins over `reRefused` for
   known transport prefixes (`lockdown|pair-record|tcc|usbmuxd`).
   At minimum, add a test row that documents the chosen behaviour
   for a TCC-style "denied" string. (`plans/2026-05-23-M5-apps-probe.md:648`)
   See `code-review.md` §"Correctness".

2. **`Save` calls `os.MkdirAll(s.dir, 0o755)` but doesn't tighten
   to `0o700`.** The probe cache lives under
   `~/Library/Application Support/ios-tidy/probes/` and contains
   device UDIDs — these are weakly sensitive identifiers. `0o755`
   is fine for a binary's data dir on a single-user Mac, but
   `0o700` is more defensible (`code-review.md` §Security:
   "is sensitive data protected appropriately?"). Low-effort fix;
   not a hard requirement. (`plans/2026-05-23-M5-apps-probe.md:1047`)

3. **`--store-dir` accepts any path with no guard against root /
   relative escape.** Per the brief's Section J: passing
   `--store-dir /` would mean `Save` calls
   `os.MkdirAll("/", 0o755)` (no-op, succeeds) then writes
   `/UDID.json` (will fail with EACCES on a non-root user, but the
   error surfaces only at write time, with a confusing message).
   For a test-only flag this is borderline acceptable, but a one-line
   guard — "reject `--store-dir` values that aren't under
   `os.UserConfigDir()`, `os.TempDir()`, or an explicit
   `IOS_TIDY_ALLOW_STORE_DIR=1`" — would prevent foot-guns.
   (`plans/2026-05-23-M5-apps-probe.md:1938`)
   See `agent-safety/SKILL.md` §Destructive Operations / Recovery
   Mindset.

4. **`TestProbe_classifyErrors_table` does not exercise the
   `connect afc service failed` case under *mixed* casing.** Row
   `{"connect afc service failed", "connect afc service failed",
   ProbeRefused}` is all-lowercase. RESEARCH.md §3 cites go-ios
   issue #653 verbatim as `"connect afc service failed"` (lower),
   but the regex is anchored as `(?i)connect afc service failed`,
   which would also match `"Connect AFC Service Failed"`. Add one
   row with mixed casing to lock in the `(?i)` behaviour against
   regressions. (`plans/2026-05-23-M5-apps-probe.md:578-580`)

### Low

1. **Task 9 admits the test will PASS on first run because Task 8
   already shipped the implementation.** Plan explicitly flags this
   ("This is a TDD-style 'watch the test pass against the
   implementation that was supposed to make it pass'"). That's
   honest, but per `agent-safety/SKILL.md` ("Tests Are Specifications")
   it would be cleaner to move the Load test code into Task 8's
   RED step, so Task 8's test drives Task 8's implementation, and
   Task 9 disappears entirely. The current shape works but isn't
   "TDD cadence" in the strict sense. (`plans/2026-05-23-M5-apps-probe.md:1222-1228`)

2. **`appsDeps.Now` is declared but never used in the live code
   path of `appsListCmd`.** Only `appsProbeCmd.run` references
   `now := c.deps.Now`. If M5 needs an injectable clock only for
   probe, lift `Now` out of `appsDeps` and into `appsProbeCmd`
   directly to narrow the surface. Minor maintainability concern.
   (`plans/2026-05-23-M5-apps-probe.md:1467`)

3. **`TestProbe_classifyErrors_table` uses `errors.New(c.errMsg)`
   inside a `t.Run` closure that references the loop variable `c`.**
   Pre-Go-1.22 this would be a classic loop-variable capture bug,
   but SHARED_CONTEXT.md §1 pins Go 1.23 so the per-iteration scope
   makes this safe. Worth a one-line `c := c` comment for readers
   familiar with the old pattern. (`plans/2026-05-23-M5-apps-probe.md:594-609`)

4. **Task 14 commit message body uses NZ English ("integration")
   correctly, but the in-plan `t.Logf` strings use `"outcome=%s
   detail=%q"` — no behaviour issue, just noting the integration
   test prints `outcome=refused` which downstream tooling may
   want to grep. Optional: stable
   `[probe] bundle=%s outcome=%s detail=%q` prefix for log
   discoverability. (`plans/2026-05-23-M5-apps-probe.md:2300,2334`)

5. **NZ English check:** searched the plan body for `behavior`,
   `organization`, `color`, `center`. None present — all NZ
   spellings (`behaviour`, `organisation`, etc.). Good.

## Classification correctness assessment

The regex set and its tests are the heart of M5. Walking each:

| Regex | Case-insensitive? | Behaviour | Verdict |
|---|---|---|---|
| `(?i)application.*not installed` → `ProbeUnknown` | ✓ (`(?i)`) | matches "Application X not installed", "application 'x' is not installed on device" | ✓ |
| `InstallationLookupFailed` → `ProbeUnknown` | ✗ (deliberate; daemon's literal code) | substring match | ✓ — plan comments cite the deliberate case-sensitivity |
| `(?i)denied\|refused\|vendcontainer.*failed\|connect afc service failed` → `ProbeRefused` | ✓ (`(?i)`) | matches `"VendContainer failed: denied"`, `"connect afc service failed"`, mixed casing | ✓ for the documented cases; ⚠ false-positive risk on incidental "denied" (Medium #1 above) |
| Anything else → `ProbeError` | n/a | default branch | ✓ |
| `errors.Is(err, context.DeadlineExceeded)` → `ProbeError` | n/a | explicit branch in `Probe` | ✓ — honours the binding "timeout ≠ refused" rule |

## Open questions raised by the plan author

1. **`internal/iosbackend/sandbox.go` partial implementation (Open
   + Close only, rest stubbed as M6).** My view: **acceptable**
   progressive delivery. The stubs return a non-nil error
   (`errNotImplementedM6`) rather than panicking — caller will
   surface a clean error if M6's code path is reached prematurely.
   M5's acceptance criteria only exercise Open/Close, so this is
   genuine scope discipline rather than gold-plating M6. The only
   risk is forgetting to lift the stubs in M6; mitigate by leaving
   a `// TODO(M6): implement` comment alongside `errNotImplementedM6`
   so M6's plan can grep for it.

2. **Classification regex case-sensitivity / coverage by
   integration test feedback.** My view: **agreed.** The plan's
   stance — "add a table row when reality contradicts the regex,
   don't widen the regex blindly" — is the right discipline.
   Captures evidence per RESEARCH.md §3 properly.

3. **`--store-dir` visibility (real, not hidden flag).** My view:
   **agreed** that a hidden flag isn't worth pulling in a flag
   library. The flag is documented honestly as "Mainly for tests"
   and Medium #3 above is a separate concern about *guarding* it,
   not hiding it. Real-and-guarded is the right combination.
