## Summary

The plan is, on the whole, an unusually careful first-destructive-milestone draft. The dry-run-never-mutates contract is encoded across multiple complementary tests (spy slice on `FakeClient`, double guard via `RemoveCalls == 0` AND `t.Fatalf` in `RemoveFn`), the plan-before-prompt ordering is asserted by capturing `stderr` at the moment of prompt entry, EOF-as-no is explicitly covered, and the integration test is double-gated on both `IOS_TIDY_TEST_UDID` and `IOS_TIDY_TEST_ALLOW_DESTRUCTIVE=1`. Stderr/stdout discipline (stdout intentionally empty) is asserted by tests.

That said, the plan has one **Critical** safety lapse â€” `--dry-run` is short-circuited only *after* `Client.List` is consulted, which is fine, but the test that asserts no `Remove` call under `--dry-run` does NOT also assert that the **prompter** was never called. The Task 8 test fixes this via `t.Fatalf` inside `ConfirmFn`, so this concern is partially addressed â€” but the cleaner enforcement (and the deeper one) is whether the `--dry-run` short-circuit lives strictly before the prompt branch in source. It does (Task 8 Step 3 inserts the check `AFTER ui.RenderPlan, BEFORE the prompt`). So the spy contract is met. The real Critical is below.

The dominant safety risks I found are: (1) the adapter's `Remove` is **not atomic with the pre-delete `List`**, so the byte-freed total can be misleading or wrong if the device state changes between snapshot and remove; combined with (2) `statCrashReport` being **left unimplemented in this plan** (`return 0, fmt.Errorf("unimplemented: â€¦")`) â€” which means the adapter as written will always return `Bytes: 0`, and the integration test in Task 14 does NOT assert `res.Bytes > 0`, so the bug is not caught. (3) The Task 11 partial-failure summary uses the locution `"1 failures."` (always plural), a minor UX wart but fine.

There are several **High** issues around plan-format compliance and TDD-cadence violations in Tasks 10â€“12, which the plan author flags up-front but does not fix. The author offers to swap order if reviewers require strict RED-first â€” reviewers should take that offer.

## Findings

### Critical

**C1. `statCrashReport` is shipped as an explicit `unimplemented` stub, which silently zeroes the bytes-freed figure in production.** Task 13, lines 1260-1265:
```go
func statCrashReport(d ios.DeviceEntry, name string) (int64, error) {
    return 0, fmt.Errorf("unimplemented: wire to M3's AFC client")
}
```
The caller swallows the error (`if sz, statErr := statCrashReport(entry, n); statErr == nil { bytes += sz }`), so `Bytes` will silently always be `0` for every real-device deletion. The integration test (Task 14) only asserts `res.Removed != 1`, NOT that `res.Bytes > 0`, so the bug is not detected. This is a correctness-of-user-facing-output failure on a destructive command, and it sits in the very task that the plan defers verification of to "M3's adapter shape". Recommendation: either (a) require Task 13 to wire `statCrashReport` to the M3 AFC client concretely with the M3 type names assumed, with a fallback to a clearly-named `errNoStatHelper` that fails the build if M3 didn't ship a usable helper; or (b) add a Task-14 assertion `if res.Bytes <= 0 { t.Errorf("expected bytes > 0 â€¦") }`; or (c) accept that the M4 adapter ships with `Bytes = 0` and update the M4 acceptance criterion to say so. The plan as drafted ships the bug.

**C2. `--dry-run` short-circuit lives after `List` but the Task 8 test does NOT assert that `len(actions) > 0` was rendered â€” it only asserts the total-footer string.** Looking at the Task 8 test, the assertions `Total: 2 files, 3.0 KB` and `Dry run â€” no changes made.` are present, and `RemoveCalls == 0` is asserted. The `ConfirmFn` is fatally guarded. This actually passes the brief's bar. **Downgraded to High** â€” see H1.
