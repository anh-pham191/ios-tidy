## Summary

M6 is the most safety-sensitive milestone in the project, and the plan takes that seriously. The probe gate, the dry-run spy assertions, the strict typed-bundle-ID gate for `--include-documents`, the `--yes` cannot bypass the strict gate property, and the per-file Documents Execute path are all present, tested, and unambiguous. The README is embedded verbatim (not a TODO) and covers the honest "what it can't do" surface as well as both Tahoe TCC (#710) and iOS 17.1 house_arrest (#653) troubleshooting items.

The single most important user-data safety mechanism — typed-bundle-ID confirmation that `--yes` does NOT bypass — is locked in by `TestAppsClean_yesDoesNotBypassDocumentsStrictGate` (Task 13, Step 7). This is the load-bearing test for the entire `--include-documents` flow and it is correct.

Two High findings concern the dry-run spy fake's completeness and the README's missing example for one subcommand. Two Medium findings concern open-question resolutions that warrant explicit reviewer agreement before execution. No Critical findings.

## Findings

### Critical

None.

### High

**H1. Dry-run spy assertion is partial — only `RemoveCalls` and `RemoveAllCalls` checked; `Close` not asserted, and Documents target with dry-run is not exercised.**

Task 11's `TestAppsClean_dryRunNeverCallsRemove` configures `WalkResults` for `tmp` and `Library/Caches` only, using the default include flags. It correctly asserts `RemoveCalls` and `RemoveAllCalls` are empty. But the brief calls for testing the FULL destructive matrix under dry-run. A separate dry-run case with `--include-documents` (where Execute would otherwise call per-file `Remove` on `Documents/*`) is not present. The spy fake only catches `Remove`/`RemoveAll`; if a future edit accidentally moves the dry-run short-circuit AFTER the Documents Execute branch, the existing test would still pass (because no Documents files are in WalkResults).

Add a second dry-run test: `--include-documents` with `Documents` in WalkResults, asserting `len(fakeFS.RemoveCalls) == 0` AND `len(prompter.AskedLines) == 0` (the strict gate must also be skipped under dry-run since dry-run announces no changes will be made). Without this the dry-run guarantee for the Documents path is not specified by a test.

Severity rationale: dry-run is the brief's binding promise ("MUST NOT call any mutating method on FS"). A partial test is a partial specification.

**H2. README missing examples for two subcommand variants required by the brief.**

The brief ("Every command has an example") is met for `devices`, `storage`, `crashlogs list/pull/clean`, `apps list/probe/clean`. But:

- The `apps clean --include-tmp --include-caches` combination (Documents off, both file-cache targets on) is not shown — and given Open Question #4's resolution that explicit flags REPLACE the defaults, this is a non-obvious case users will hit.
- `apps probe --bundle ID` form (single-bundle, not `--all`) is shown but `apps probe --bundle X --bundle Y` (the brief's exact signature) is shown only once in §`apps probe`; that's fine, retract this part.

Add the `--include-tmp --include-caches` combined-explicit example to the `apps clean` examples block to lock in the Open Question #4 resolution in user-facing docs.

### Medium

**M1. Open Question #2 (case-sensitive typed-bundle-ID) and Open Question #4 (explicit flags REPLACE defaults) are reasonable but reviewer should explicitly assent.**

Both resolutions are defensible. Case-sensitive comparison after `TrimSpace` is the safer of the two options (it refuses lenient matches that could slip a typo past the gate). Explicit-flags-replace-defaults matches the brief's hint and is the principle-of-least-surprise reading once you've named one target explicitly. The plan reasons both correctly. I concur with both as written. Surfacing as Medium so the user knows these are reviewed-and-accepted, not slipped-through.

**M2. `ui.Prompter.ReadLine` introduced in M6 rather than M3.**

The plan adds `ReadLine` to `ui.Prompter` in Task 13. This is the only new seam extension in the plan and it is justified — M3/M4 never needed line input, and `Confirm`'s yes/no is genuinely a different shape. Adding it in M6 keeps M3/M4 unchanged, which respects "Tests Are Specifications" — M3/M4's existing `FakePrompter` tests should still pass because the new methods are additive. The plan correctly:

1. Adds the method to the interface (Task 13 Step 3),
2. Updates the real `stdinPrompter` impl (Task 13 Step 3),
3. Extends `FakePrompter` with `Lines`/`AskedLines`/`ReadLineErr` (Task 13 Step 3),
4. Writes unit tests for `FakePrompter.ReadLine` (Task 13 Steps 1–2).

All four boxes ticked. The only minor concern: extending an existing interface method-set is technically a breaking change for any out-of-tree impl. There are none in this project, so this is purely a forward-design observation rather than a blocker.

**M3. `executeRemoveAll` accounts via plan totals, not via post-RemoveAll verification.**

Task 4's `executeRemoveAll` (lines 622–632) trusts the pre-walk plan accounting after a successful `RemoveAll`. The comment in the plan acknowledges this is an approximation: "If the device's view changed between Walk and RemoveAll, the integration test will catch a mismatch". This is fine for tmp/Caches — the worst-case outcome is over-reporting bytes freed by a small amount. The behaviour is consistent across the unit tests. Not a finding to block on; flagged so the reviewer can confirm acceptance.

**M4. `Sandbox.FakeSandbox.OpenResults` key encoding (`"U1\x00com.example.app"`) is asserted but not introduced by this plan.**

The plan repeatedly uses `map[string]sandbox.FS{"U1\x00com.example.app": fakeFS}` as the `FakeSandbox.OpenResults` keying, but the encoding scheme (`udid + \x00 + bundleID`) is M5's territory. The plan acknowledges: "If FakeSandbox.OpenResults' key encoding is different in M5's implementation, adjust." That's acceptable, but the executing agent must verify the M5 encoding before running these tests rather than blindly copying. Add a verification micro-step at Task 1 ("read internal/sandbox/fake.go to confirm FakeSandbox.OpenResults key format") to surface the dependency.

### Low

**L1. Task 18 ("Coverage sweep") contains the single permitted `TBD` for the per-package gap list.**

The plan's self-review §2 explicitly notes this and bounds it with a concrete procedure (run cover, identify red, add ONE test per branch, commit per package, never modify existing tests). The "tests are specifications" rule is explicitly invoked. Acceptable as written.

**L2. Task 14 ("Open-after-stale-probe error path") tests a hint message string match.**

`!strings.Contains(stderr.String(), "ios-tidy apps probe --bundle com.example.app")` is a brittle assertion. If a future copy-edit changes the wording it will silently break this safety hint test. Consider matching just `"apps probe"` and `"stale"` separately. Minor — the current assertion is defensible because the exact command line is what users would copy-paste.

**L3. Plan time estimates are aggressive.**

5 hours of agent work for 19 tasks at the granularity described is optimistic given the verify-RED + verify-GREEN runs. Not a blocker.

## Open questions raised by the plan author

1. **Dry-run + Open reconciliation.** Author's resolution: dry-run opens + walks + stats (read-only), then prints plan and "Dry run — no changes made.", never calls Remove/RemoveAll. **Reviewer view: agree.** A "static" dry-run that doesn't open the sandbox would give the user a target list without byte totals, which defeats the dry-run's job. The spy fake correctly enforces no mutating calls. The only addition I would request is the missing Documents-path dry-run test (H1).

2. **Case-sensitivity of typed-bundle-ID confirmation.** Author's resolution: strict case-sensitive `==` after `TrimSpace`. **Reviewer view: agree.** Lenient matching opens an avenue for autocomplete typos. The user typed the bundle ID themselves on the CLI moments ago — they know its case.

3. **`Failure` type re-use vs duplication.** Author's resolution: define a fresh `sandbox.Failure` rather than importing from `crashlogs`. **Reviewer view: agree.** Sideways package dependencies between peer domain packages add coupling for two lines of saved duplication. Local definition is correct.

4. **`--include-documents` solo behaviour.** Author's resolution: any explicit `--include-*` flag REPLACES the defaults. **Reviewer view: agree.** Matches the brief's hint and principle of least surprise. Worth adding the explicit-multi-flag example to the README (H2).

5. **Integration test `Push` availability.** Author's resolution: test code reaches into the iosbackend package and calls raw go-ios `Push` directly; does NOT extend `sandbox.FS`. **Reviewer view: agree.** Push has no production use case in M6, and adding it to the seam interface would force every consumer (and every fake) to implement a method only the integration test uses. Test-code escape hatch is the right call.
