# M2 plan — review cycle 1

**Reviewer:** independent peer-reviewer subagent
**Plan reviewed:** docs/superpowers/plans/2026-05-23-M2-storage.md
**Date:** 2026-05-23

## Summary

The plan is thorough, well-structured, and covers every M2 acceptance criterion in
`SHARED_CONTEXT.md` §8 with concrete code, named seam types, and bite-sized tasks.
Seam adherence is excellent: `storage.DeviceInfo` and `apps.App`/`Lister`/`Uninstaller`
are reproduced verbatim, the go-ios import fence is respected (only `internal/iosbackend/`
imports go-ios), and the four open questions at the top are surfaced clearly.

However, two concrete correctness/safety issues block confident execution. First,
the production adapters in Task 10 call `defer client.Close()` on the go-ios
`afc.Client` and `installationproxy.Connection`, but neither `Close()` method appears
in the verbatim go-ios signatures pasted in `RESEARCH.md` §2 — the plan never verifies
those methods exist, so as written the package may not compile. Second, `fetchInParallel`
in Task 11 step 4c does not honour `ctx` cancellation: if the caller cancels mid-flight,
both goroutines run to natural completion before `wg.Wait()` returns, contradicting the
"first-error captured, both goroutines complete or ctx cancelled" claim in Open
question 4. There is no goroutine leak (WaitGroup is correct), but cancellation is
purely cosmetic.

Several Medium issues relate to RED steps that pass immediately because earlier
skeleton code already covers them (a self-acknowledged TDD departure), and to
non-trivial `mapAppInfo` translation logic in Task 10 receiving no unit test
despite being a pure function. Otherwise the plan is in good shape; recommend
Revise (small, focused changes) and re-review.

## Findings

### Critical

**C1. `afc.Client.Close()` and `installationproxy.Connection.Close()` are called but not verified in RESEARCH.md** — `internal/iosbackend/storage.go` and `internal/iosbackend/apps.go` (Task 10).

Quote: `defer client.Close()` (storage.go) and `defer conn.Close()` (apps.go).

Rule violation: `SHARED_CONTEXT.md` §7's "References to types/functions/methods not defined in this plan or in SHARED_CONTEXT.md §3" forbidden-pattern list, plus the plan's own self-review claim "go-ios signatures verified via `gh api ...`: `afc.New`, `(*afc.Client).DeviceInfo`, ... `installationproxy.New`, `(*installationproxy.Connection).BrowseUserApps`, `(*installationproxy.Connection).Uninstall`." `Close` is conspicuously absent from both that list and from the verbatim signature dump in `RESEARCH.md` §2 (which enumerates `New`, `NewFromConn`, `List`, `Stat`, `WalkDir`, `Remove`, `RemoveAll`, `Pull`, `PullSingleFile`, `Push`, `DeviceInfo` for AFC, and `New`, `BrowseUserApps`, `BrowseSystemApps`, `BrowseFileSharingApps`, `Uninstall` for installationproxy).

Concrete fix: before executing Task 10, the executor should `gh api repos/danielpaulus/go-ios/contents/ios/afc/client.go` and `.../installationproxy/installationproxy.go` to confirm whether `Close()` exists. If it does not, replace `defer client.Close()` with the actual cleanup primitive go-ios exposes (typically closing the underlying `ios.DeviceConnectionInterface`). If `Close()` does exist, update the self-review's signature list to include it so the next reviewer doesn't have to re-verify. Either way, this is **gating** — `go build ./internal/iosbackend/...` (Task 10 step 4) will hard-fail if the methods don't exist.

### High

**H1. `fetchInParallel` does not honour `ctx` cancellation** — `cmd/ios-tidy/storage.go` (Task 11 step 4c).

Quote (Open question 4): "Reviewer can argue this is over-engineering for two calls and we should just run them serially; I keep the parallel form because the prompt explicitly asks for it." Quote (helper doc): "We do this with a sync.WaitGroup + first-error capture instead of pulling in golang.org/x/sync/errgroup".

The implementation:

```go
wg.Add(2)
go func() { defer wg.Done(); info, infoErr = sc.DeviceInfo(ctx, udid) }()
go func() { defer wg.Done(); appList, appsErr = al.UserApps(ctx, udid) }()
wg.Wait()
```

Rule violation: `code-review.md` "Correctness" — the plan's own framing ("first-error captured, both goroutines complete or ctx cancelled, no leak", per the review brief §H) is not met. If the caller's `ctx` is cancelled, neither goroutine is signalled — they each block until their downstream call returns naturally. `errgroup.WithContext` was forbidden in the prompt precisely so the plan would build this itself; instead the plan dropped the cancellation half of the contract without saying so.

Concrete fix: either (a) honestly downgrade the helper to "run both in parallel and wait for both; ctx is forwarded but not used to short-circuit," and acknowledge that's the cost of avoiding errgroup; or (b) implement the missing piece: derive a child `ctx, cancel := context.WithCancel(ctx)`, call `cancel` from the first goroutine that errors, and rely on the underlying `sc.DeviceInfo`/`al.UserApps` to surface `ctx.Err()` promptly. Add a test (e.g. `TestStorageCmd_contextCancellation_aborts`) that injects a fake whose `DeviceInfo` blocks on `<-ctx.Done()`. Without that test, this code path is unverified.

**H2. `mapAppInfo` is a pure function with non-trivial fallback logic and zero direct unit tests** — `internal/iosbackend/apps.go` (Task 10 step 2).

Quote: the function picks `CFBundleName` falling back to `CFBundleDisplayName`, picks `CFBundleShortVersionString` falling back to `CFBundleVersion`, and routes every numeric key through `asUint64`. None of this fallback logic is unit-tested — only `asUint64` is (Task 9), and the integration test in Task 12 only asserts "BundleID non-empty".

Rule violation: `SHARED_CONTEXT.md` §5 ("aim for 80%+ on `internal/*` excluding `internal/iosbackend/`") — `iosbackend` is exempt in general, but `mapAppInfo` takes a plain map argument and is fully testable without a device. Plus `code-review.md` "Testability": "Pure functions ... tested as pure (no fakes)".

Concrete fix: add a Task 10 sub-step writing a `TestMapAppInfo` table-driven test (file: `internal/iosbackend/apps_test.go`, additive). Cases: bundle id present; name uses `CFBundleName` when present; name falls back to `CFBundleDisplayName` when `CFBundleName` missing/empty; version fallback path; `UIFileSharingEnabled` true and false; `DynamicDiskUsage` arriving as int64/float64/string (delegating to `asUint64`); missing container yielding empty string. Without these the only verification of the translation layer is a real device run, which contradicts the M2 AC bullet about 100% unit-test coverage of the consumer-facing logic.

### Medium

**M1. Several "RED" steps pass immediately because the prior skeleton already covers them.** Steps 2b, 3b, 5b, 6b, 7b explicitly state "passes immediately" and the plan calls it "the rare … acceptable" case. Per `SHARED_CONTEXT.md` §5 ("Every step that adds production code is preceded by a step that writes a failing test … Show the expected failure message in the plan"), a test that does not fail before its implementation is not a RED step. Recommend either (a) restructure Task 11 so each test fails before its corresponding skeleton update, or (b) collapse the steady-state tests into a single "characterise current behaviour" task and stop labelling them as RED→GREEN cycles. Either way the plan currently presents passing tests as if they were verifying TDD discipline; reviewers downstream will be confused.

**M2. Tasks 1 and 3 add production type declarations with no preceding RED step.** The plan justifies this with "type declarations only; covered downstream by Tasks 2 and 11." This is a pragmatic call, and the types are verbatim from `SHARED_CONTEXT.md` §3 so there's no design discretion to test. Flagging for the human reviewer rather than insisting on a workaround — but the strict reading of §5 is that even `package storage` + struct declaration counts as production code and should have a RED step. If the human reviewer agrees with the pragmatic interpretation, the §5 wording in SHARED_CONTEXT could be tightened in a follow-up.

**M3. JSON output shape is asserted via partial-shape decoding, never via a full golden assertion.** Test step 5a decodes into an anonymous struct exposing only `Model`, `AFCTotalBytes`, `AFCFreeBytes`, `BundleID`, `DynamicBytes`, `StaticBytes`. Other documented fields (`afcBlockSize`, `name`, `version`, `container`, `fileSharingEnabled`, `applicationType`) are not asserted. Per the review brief §I ("If the JSON shape isn't fully shown in a test, that's Medium") this is Medium. Recommend extending the test to assert every camelCase key.

**M4. Task 10 step 2's note "in Go you cannot have two `import` blocks at the same top level" is factually wrong.** Go allows multiple top-level `import` declarations; `gofmt` will merge them. This doesn't change the executor's actions (one block is still the correct outcome) but the rationale will confuse anyone who knows the language. Recommend rewriting the parenthetical as "gofmt collapses multiple top-level `import` declarations into one; group them yourself to avoid a re-format diff."

**M5. The "ctx cancellation" comment in `internal/iosbackend/storage.go` is misleading.** Quote: "ctx is honoured by closing the AFC connection if the caller cancels." The code shown does not wire `ctx` into the AFC client at all — it only checks `ctx.Err()` twice before and between blocking calls. Either delete the misleading sentence or wire `ctx` to the underlying socket (which go-ios may not expose). Same comment issue in `apps.go`.

**M6. Open question 3 (`afcTotalBytes` vs `totalBytes`) is well-flagged, but the chosen naming bleeds into the test fixtures and assertions throughout Task 11.** If the reviewer prefers the bare-key form, the plan requires touching at least five places (struct tags, test decoder struct in step 5a, comment in Task 1, etc.). Not a defect, but worth noting that "swap the struct tags in Task 1" understates the actual scope. Recommend the executor confirm the naming choice with the human before starting Task 1, so the fix isn't a multi-file find-replace later.

### Low

**L1.** Task 11 step 4c's `fetchInParallel` uses unnamed return-value pattern with named outer variables (`var info ...; var appList ...`). Idiomatic Go for this case is a small struct or named returns; current form is fine but mildly noisy.

**L2.** `FormatBytes` test case `{"binary kilobyte stays kilobyte", 1024, "1.0 KB"}` — value `1024/1000 = 1.024` formatted with `%.1f` is `"1.0 KB"` (correct rounding), but the test name suggests intent worth a code comment in `bytes.go` documenting "we use SI not IEC".

**L3.** `internal/iosbackend/apps.go` repeats the `ios.GetDevice` + `installationproxy.New` + `defer conn.Close()` pattern in both `UserApps` and `Uninstall`. Future M5/M6 adapters will repeat it again. A tiny helper `withConn(udid, fn func(*installationproxy.Connection) error) error` would dry this up; not required for M2 but worth flagging.

**L4.** Commit subject "feat: wire storage subcommand with device/json/limit flags" is 63 chars — fine. All subjects scanned are ≤72 chars and imperative. ✓

**L5.** Task 13 step 4 expects `./ios-tidy storage --help` to "exit non-zero (per `flag` conventions)." This is true for `flag.ContinueOnError` + `--help`, but worth a parenthetical pointing at `flag.ErrHelp` so the executor doesn't treat the non-zero exit as a failure.

## Open questions raised by the plan author

**OQ1. `ui/bytes.go` ownership (M1 vs M2).** Author places it in M2 and notes M1 may already own it. **My view:** correct call. M1's acceptance criterion does not require byte formatting (M1 emits device name/model/iOS version/UDID/trust state, none of which are byte counts), and M2 explicitly requires "byte-formatting" coverage. Keep in M2. If a future review of M1 reveals it was already created there, this task collapses to a no-op as the author predicts.

**OQ2. `ui/table.go` ownership.** Author assumes M1 created a minimal `Table` and Task 6 extends it. **My view:** correct, with one caveat — the plan should not silently rewrite M1's `Table`. Task 6's instruction "If M1's `Table` already exists with a different shape, do not rewrite it — add only the `Alignment` enum and a new `Render` method" already says this; reinforce it during execution.

**OQ3. `afcTotalBytes` vs `totalBytes` JSON keys.** Author chose AFC-prefixed names; `SHARED_CONTEXT.md` §10 caveat 4 explicitly endorses this. **My view:** keep the AFC prefix. The author's "swap the struct tags in Task 1" understates the scope — see M6 — but the choice itself is correct.

**OQ4. errgroup substitute (`runPair`/`fetchInParallel`).** Author retains parallel form. **My view:** the parallel form is fine *if* the cancellation contract is honoured (Finding H1) and a test exists. As written, the goroutine block runs to completion regardless of context — neither parallel nor serial, just "concurrent without responsiveness". Address H1 before relying on this code path.

## Recommendation

**Revise and re-review.** Gating findings: C1 (`Close()` not verified — risk of build failure on Task 10) and H1 (concurrency contract not delivered). H2 and M1 are strong-encourage but not strictly blocking. The plan is mostly sound and a single targeted revision should clear it for execution.
