**Reviewer:** independent peer-reviewer subagent

## Summary

The plan is thorough, well-organised, and demonstrably backed by SHARED_CONTEXT.md §3 (seams used verbatim) and RESEARCH.md (API signatures cited with source URLs in Task 8). Every acceptance criterion in §8 M1 maps to at least one concrete task, every code block is shown in full, and commit safety is honoured (explicit paths, explicit "wait for user approval" lines). The author also surfaces five open questions at the top and explains every deviation from strict TDD cadence rather than burying them.

That said, there are two issues serious enough to revise before execution. First, the real `trustChecker` adapter in Task 8 collapses every error from `ConnectLockdownWithSession` — including legitimate transport failures — into `(false, nil)`, which contradicts the seam's own documented contract ("only transport failures should produce a non-nil error") and the M1 acceptance criterion ("Returns non-zero exit on transport failure"). Second, the planned "transient red" window between Tasks 5 and 7 means the verify-RED steps for Tasks 5 and 6a–6e never actually run before their implementations land; the test files compile-fail with `missing function main` rather than producing the specific `undefined: newDevicesCmd` / behavioural mismatch failures the plan claims to expect. There are also a handful of medium issues (lock-in-after-the-fact tests in Task 3b/3c/3d, Tahoe TCC stderr hint promised in OQ#5 but never implemented in code, and the metadata-skew note from RESEARCH.md §7 for M2 leaking into the trust path here).

Recommendation: **Revise and re-review.** The fixes are small and local.

## Findings

### Critical

None. No safety-rule violation, no `git add -A`, no test-modification-after-the-fact, no destructive operations in M1.

### High

**H1. `trustChecker.Trusted` swallows transport errors, contradicting the seam contract and the M1 transport-failure acceptance criterion.**

Quote (Task 8 Step 2, real implementation):
> ```go
> conn, err := ios.ConnectLockdownWithSession(entry)
> if err != nil {
>     // Untrusted, pairing-pending, denied, or missing pair
>     // record all land here. We deliberately do NOT return the
>     // error: from the CLI's perspective these are all "the
>     // device is reachable but not trusted", and a non-nil err
>     // would force a non-zero exit (acceptance criterion:
>     // only transport failures should fail the command).
>     return false, nil
> }
> ```

Rule violated:
- SHARED_CONTEXT.md §3 (plan's own interface doc): "It is allowed to return (false, nil) for 'untrusted but reachable' — only transport failures should produce a non-nil error."
- SHARED_CONTEXT.md §8 M1 acceptance: "Returns non-zero exit on transport failure".
- RESEARCH.md §6: distinguishes `PairingDialogResponsePending`, `UserDeniedPairingError`, and missing pair-record (all "untrusted") from transport errors (usbmuxd dead, etc.). The plan's own Open question #3 also acknowledges this distinction exists.

`ConnectLockdownWithSession` returns errors for at least two distinct classes: trust-state errors (untrusted, pending, denied, missing pair record) and transport errors (usbmuxd socket gone, device unplugged mid-call, TCC pair-record read blocked on macOS 26 Tahoe per RESEARCH.md §6). Collapsing all of them to `(false, nil)` means a real transport failure — e.g. usbmuxd dying between `ListDevices()` and the per-device probe — will be silently reported as "untrusted" and the CLI will exit 0. This breaks the M1 acceptance criterion as written and breaks the seam contract Task 2 declares.

The unit test `TestDevicesCmd_trustCheckFailureReturnsNonZeroExit` (Task 6e) only verifies the handler reacts correctly when `FakeTrustChecker.Err != nil` — it does not (and cannot) verify the real adapter ever returns a non-nil error.

Recommended change: classify errors in `trustChecker.Trusted`. At minimum, distinguish "no pair record / pairing dialogue / user denial" (return `false, nil`) from anything that looks like a transport failure (return `false, err`). The pragmatic approach is a string-match on the well-known error strings from `go-ios/ios/lockdown_value.go` and `connect.go`. Either implement that classification in Task 8a, or weaken the seam doc-comment in Task 2 to match the adapter's actual behaviour and add a test in Task 6 covering the resulting "any error returns false, nil" contract — but the first option is the one that actually meets the acceptance criterion.

**H2. The "transient red" window between Tasks 5 and 7 means several verify-RED steps never run; the plan's RED → verify-RED → GREEN cadence is broken for the most important production code in M1.**

Quote (Task 6a, Step 4):
> Expected: FAIL with `missing function main` — the package still doesn't compile until Task 7 adds `main.go`. This is the same transient red noted in Task 5 Step 4. Skip to Task 6b; we'll re-run all `cmd/ios-tidy` tests at the end of Task 7.

Quote (Task 6b, Step 6):
> ```bash
> # We cannot run yet — Task 7 makes the package compile.
> ```

Quote (Task 6c, 6d, 6e): tests are written but never run individually; Task 6 Step 10 explicitly defers all execution to Task 7's final sweep.

Rule violated:
- SHARED_CONTEXT.md §7 "Self-review checklist", item 4: "every production-code step is preceded by RED + verify-RED steps and followed by verify-GREEN".
- SHARED_CONTEXT.md §5: "Every step that adds production code is preceded by a step that writes a failing test and a step that verifies the failure. Show the expected failure message in the plan."

The verify-RED for Task 6a is supposed to confirm the test fails *for the specific reason the test exists* (`undefined: newDevicesCmd`). Instead the test file fails to compile at all because `package main` is incomplete — the same compile error would mask any subsequent test the author writes. A future executor wouldn't learn anything about whether the test's assertion is correctly formulated until the very end of Task 7, by which point five tests and three large production files are all landing together. If any of those tests is mis-asserting (e.g. a typo in `"udid": "AAAA"` vs the encoder's actual output), the failure surfaces in a green sweep rather than one isolated red.

Recommended change (any of these; first is cleanest):
1. Reorder: put Task 7 (`main.go` + `dispatch` skeleton + `iosbackend` stubs) *before* Task 6. The dispatcher and `iosbackend` constructors are pure plumbing; with them in place, every `devices_test.go` test in Task 6 will produce a clean, isolated `undefined: newDevicesCmd` red, then a clean per-test green. The compile-time interface assertion already in Task 7a can stand alone.
2. Alternatively, fold the `version.go` definition and a stub `main()` into Task 5 so the package always compiles, then build out `dispatch` and `devicesCmd` incrementally. The "Version has non-empty default" test can then verify-RED with `undefined: Version` and verify-GREEN as a real isolated event.

Either reorder restores the RED → verify-RED → GREEN cadence for what is the single most important behavioural file in M1.

### Medium

- **M1. Open question #5 (macOS Tahoe TCC #710 stderr hint) is described in the OQ section but never appears in any task's code.** The OQ promises "M1 surfaces it as a `stderr` hint only when the trust check returns an error mentioning `pair record` or `permission denied`", but the trust check in Task 8 returns no error in those cases (per finding H1) and no code path inspects an error string. Either implement the hint (a `fmt.Fprintf(c.errOut, ...)` in `devicesCmd.Run` when the trust check returns an error containing those substrings) or strike OQ #5. Currently it is a promise the plan does not keep.

- **M2. Tasks 3b, 3c, 3d "write the failing test" steps are tests of behaviour already implemented in 3a's `fake.go`.** They are honest about this ("the implementation from Step 3 already supports …"), but the RED step is performative — the test passes immediately. Per SHARED_CONTEXT.md §7 the RED gate is supposed to be a real failure. These are tolerable as "lock-in specifications" (the plan says so) and they ARE valuable as regression guards, but they are not TDD. Two options: (a) merge them into 3a so the test file is written once with all four cases up front before fake.go exists, all four go red together, then fake.go makes them all green; or (b) keep them but stop calling them RED — rename the steps "lock-in test" so future executors don't think strict TDD was honoured here.

- **M3. `TestDevicesCmd_tableShowsTrustStateAndMetadata` (Task 6a) asserts on output containing the *literal* substring `"BBBB  -"`** — this couples the test to the table renderer's exact two-space gutter from `ui/table.go`. If anyone later widens the gutter or right-pads with tabs, the test fails for a cosmetic reason. Prefer asserting on the row's cells (split the table line on whitespace, check `cells[1] == "-"`) or assert on the row containing `"BBBB"` and the same line containing `"-"` and `"untrusted"`. Same issue, less severe, in the `TestRenderTable_*` table-renderer tests, but those are testing the renderer so the literal-output couple is correct there.

- **M4. The JSON output uses snake_case (`"ios_version"`).** No convention in SHARED_CONTEXT.md mandates this either way — but it is a public surface decision being made silently. Worth surfacing as an open question, or at least a note: "JSON keys are snake_case for grep-friendliness." Future milestones (M2 `--json` for storage, M5 for probe) will inherit the choice.

- **M5. `readMetadata` in Task 8 silently drops errors from `ios.GetValues`.** The justification is fine (an untrusted device legitimately can't be read), but a transient transport error from `GetValues` for a *trusted* device would also be silently dropped, producing a device row with blank fields. Either log the error to stderr at warning level, or — better — return the error and let the caller decide. RESEARCH.md §7 question 4 (`DynamicDiskUsage` populated only after recent launch) is a sibling: cold devices can produce empty values too. This is M1's only correctness corner that isn't already tested.

- **M6. The `TestDispatch_devicesSubcommandWiresFakes` test passes `[]string{"devices"}` to dispatch with a FakeLister holding one device.** The handler will succeed and print AAAA. But the test doesn't pass `--json`, so it's actually re-testing what Task 6a already covers (table rendering for a single device). It's wiring-coverage, not behaviour-coverage. The wiring is the point of the test, so this is fine — but consider asserting only on `lister.Calls == 1` and `exit == 0`, not on output content; the output content is Task 6a's domain.

- **M7. `dashIfEmpty` is private to `cmd/ios-tidy` but the `ui` package is the logical home for it** (it's a presentation helper, not a CLI concern). M2 onward will need the same "show '-' for blank" logic for the storage view and the apps table. Consider promoting it to `internal/ui` in this plan; defer if you prefer to grow the ui package one helper at a time. Not blocking.

Trace it carefully when implementing; the expected string in the test may be off by one space. (I haven't run it; flagging for verification.)

## Open questions raised by the plan author

The author surfaced five open questions at the top of the plan. My take on each:

1. **`ui/bytes.go` deferred to M2.** Material? Yes (it's a scope-discipline question). Needs settling before execution? Yes, but the answer is trivially yes-defer — bytes formatting has no consumer in M1. Recommended resolution: confirm defer; remove from M1's "file map" expectations entirely (which the plan already does).

2. **First commit has no pre-failing test.** Material? Yes. Needs settling before execution? Yes. Recommended resolution: approve. The TDD skill explicitly allows scaffolding exceptions, and Task 1 is purely module metadata. The wording invocation is clean.

3. **Trust-check semantics (`ConnectLockdownWithSession` as probe).** Material? Yes — this defines what "trusted" means observationally. Needs settling? Yes. Recommended resolution: accept the pragmatic choice **and** add the error-classification logic from finding H1 so the seam contract is honoured. The OQ as written is incomplete: it says the implementation returns "(false, nil) on any error" but the seam doc says only transport errors return non-nil. Pick one and make both match.

4. **Untrusted-device placeholder character (`"-"`).** Material? Low — it's a presentation detail. Recommended resolution: keep `"-"`. Rationale: it's grep-friendly, it's unambiguous (a single character in a known column position), and it survives JSON: the JSON path uses empty strings rather than dashes, so the user-facing string never appears in machine output. Approve.

5. **macOS Tahoe TCC stderr hint.** Material? Medium — it is the user's most likely first-run blocker per SHARED_CONTEXT.md §10. Needs settling? Yes. Recommended resolution: implement it in M1 (see finding M1) rather than deferring to M6 README. A one-line `if strings.Contains(err.Error(), "pair record") || strings.Contains(err.Error(), "permission denied") { fmt.Fprintln(c.errOut, "hint: macOS Tahoe TCC may be blocking pair-record reads; see go-ios issue #710") }` in the lister-error path is sufficient. This converts an OQ promise into actual code and ties M1 cleanly to the §10 sanity caveat.
