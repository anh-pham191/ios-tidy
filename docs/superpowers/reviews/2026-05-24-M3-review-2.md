## New material findings (cycle 2 only)

### Medium

1. **[Medium] Happy-path summary line silently drops the `skipped 0` literal because of format-string asymmetry.** (Task 7 Step 3, line 1739, vs Task 7 Step 1 test, line 1638.) The happy-path `fmt.Fprintf` reads:

   ```go
   fmt.Fprintf(stdout, "pulled %d of %d, %s, skipped 0, failed %d\n",
       res.Pulled, len(entries), ui.FormatBytes(uint64(res.Bytes)), len(res.Failures))
   ```

   But the abort path (line 1725) and the total-error path (line 1735) omit the bytes segment:

   ```go
   fmt.Fprintf(stdout, "pulled 0 of %d, skipped %d (declined), failed 0\n", ...)
   fmt.Fprintf(stdout, "pulled 0 of %d, skipped 0, failed %d\n", ...)
   ```

   Three different summary shapes for three code paths is the opposite of the "uniform summary shape" the Revision history claims at line 38 ("the field exists so the summary line is uniform with the abort path's output"). The happy-path test only asserts `strings.Contains(out, "pulled 2 of 2")`, `strings.Contains(out, "skipped 0")`, `strings.Contains(out, "failed 0")` — passes even with the inconsistent format because `"skipped 0"` is a substring. But a scripted consumer parsing the summary will see the bytes segment appear/disappear unpredictably.

   **Fix:** use a single format string across all three paths, e.g. `"pulled %d of %d (%s), skipped %d (declined), failed %d\n"`, with `0` and zero-byte segments filled in where appropriate. Five-minute change inside Task 7 Step 3; update the abort and error-path `Fprintf` lines to match the happy path's bytes segment. Update one test assertion if needed.

   Severity Medium because (a) the inconsistency contradicts the plan's own stated UX intent and (b) the test currently passes on the inconsistent output, so a future regression would not be caught.

### Low

2. **[Low] Task 6 Step 3 mutually-recursive task ordering.** `runCrashLogs` in Task 6 dispatches to `runCrashLogsPull` (line 1436), which exists as a stub returning exit 2 (lines 1446–1449). Task 7 then replaces the stub. This is correct and the plan handles it well, but the stub commit (Task 6 Step 11, line 1567) will land in the repo with `runCrashLogsPull` returning `2` and a "not implemented yet" stderr message. If the user runs `git bisect` between the Task 6 and Task 7 commits and lands on Task 6, `ios-tidy crashlogs pull` will be visibly broken. This is the standard cost of multi-step TDD and acceptable; mention is for completeness only.

3. **[Low] Pre-list round-trip in `Pull` is now double work.** The adapter's `Pull` calls `c.List(ctx, udid, pattern)` to populate `Pulled` and `Bytes` (Task 5 Step 3 line 1119) **and** the cmd layer calls `deps.Client.List` first (Task 7 Step 3 line 1688) for the pre-scan. So the device is walked twice on every `pull` invocation: once by the cmd-layer pre-scan's `List`, once by the adapter `Pull`'s internal `List`, plus the actual `DownloadReports` walk. That's three walks per `pull`. Two of them are cheap (`ListReports` paths only + N per-path `Stat`), but it's worth flagging. **Mitigation idea (not blocking):** have `Pull` accept a pre-computed `entries []Entry` slice via an optional path, or have `runCrashLogsPull` pass through the entries it already has. Either is a small seam tweak post-M3. Calling out so the M4 author has the option to address it when they touch this surface.

4. **[Low] Task 7 Step 1's instruction to add imports `"os"` and `"path/filepath"` in the test file before they're needed.** (Lines 1601–1603.) The note at line 1647 ("If the implementer prefers to add the imports in Step 5 instead of here, that also works") gives the implementer flexibility — fine. But Step 1's first happy-path test does NOT use `os` or `filepath`; only Step 5's overwrite test does. Adding them in Step 1 will cause `goimports`/`gofmt` to remove them before Step 5 lands. Minor friction; the implementer note covers it. Style only.

5. **[Low] Adapter's `List` swallows `Stat` errors silently.** (Task 5 Step 3 lines 1101–1106.) If `Stat` fails on a path (e.g. file deleted between `ListReports` and `Stat`), the entry is included with `Size=0` and no warning. The doc comment explains the choice ("the user can still see and pull it"). Defensible but arguably worth a stderr warning when the consumer is the cmd layer. Not blocking.
