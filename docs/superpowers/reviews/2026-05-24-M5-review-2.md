### Low

1. **`validateStoreDir` rejects `dir == "."` (root match) but
   accepts a path that is *exactly* the root.** The branch
   `if rel == "." { return nil }` after the prefix check would be
   redundant, because the prior `if abs == rootAbs { return nil }`
   already handles the equal-paths case. The current code is
   correct, but the `rel != "."` predicate inside the
   `!strings.HasPrefix(rel, "..")` arm is dead code — `filepath.Rel`
   from `/tmp` to `/tmp` returns `"."`, which the
   `!strings.HasPrefix(rel, "..")` check accepts (it's not `..`), so
   the `rel != "."` excludes that exact case, forcing the engineer
   to rely on the subsequent `if abs == rootAbs` fall-through. Works
   correctly, but the `rel != "."` is the wrong shape — if the
   intent is to require descent INTO the root (not equal-to-root),
   the explicit `abs == rootAbs` branch is the right place for it
   and the inline `&& rel != "."` should go. Pure stylistic; behaviour
   is correct. (Lines 2152-2158.)

2. **`TestAppsProbe_storeDirRejectsUnsafePath` only tests `"/"`.**
   The interesting adversarial cases — `/etc`, `~`, a relative
   `../etc`, a path that *starts* with `/tmp` but isn't under it
   (e.g. `/tmpwhatever`) — would prove the `filepath.Rel` +
   `HasPrefix(rel, "..")` logic isn't fooled by string-prefix
   coincidence. A single extra row like `"/tmpfoo"` (assuming
   `/tmpfoo` is not under `/tmp`) would lock in the absolute-path
   semantics. Not gating; the current test proves the obvious case.
   (Line 1921-1941.)
